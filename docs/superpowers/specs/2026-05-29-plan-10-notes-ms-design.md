# Plan 10 — Notes Sample Microservice

**Date:** 2026-05-29
**Status:** Approved design — implementation lives in `docs/superpowers/plans/2026-05-29-plan-10-notes.md`.

## Goal

Demonstrate the icore stack end-to-end with a small but realistic domain feature. Adds a `notes` microservice (single collection, owner-scoped CRUD) wired to the configured `DBStrategy`, gateway routes guarded by `AuthGuard` + `AbilityGuard`, and a `/_dashboard/notes` route in all three client templates. After this plan a consumer can clone icore, pick `--auth=firebase --db=supabase`, and immediately see notes flowing through the entire stack.

## Why a sample feature

icore is infrastructure. Without a demo feature, consumers have to invent one before they can verify the box is wired correctly. Notes is the smallest useful CRUD: one collection, one ownership invariant, two list filters (mine vs all-admin), one update field. It exercises:

- `DBStrategy` (read, set, update, delete, list with where-clause)
- `AuthStrategy` (gateway `req.user` populated)
- CASL `defineAbilitiesFor` (admin sees all, user sees own)
- TanStack Query (cache invalidation on mutation)
- `setNotifier` (toast on save / delete)
- i18n keys across en/ru/he
- All 3 UI libraries (shadcn / antd / mui)

## Domain shape

```ts
interface Note {
  id: string; // generated by the MS
  ownerId: string; // populated server-side from req.user.id; never trusted from client
  title: string; // 1..200 chars
  body: string; // 0..10_000 chars
  createdAt: string; // ISO; populated server-side
  updatedAt: string; // ISO; bumped on every set/update
}
```

Single collection: `notes`. No indexes beyond `ownerId`. Pagination is offset-based for simplicity (`limit + offset`).

## Architecture

```
              client (logged in)
                   │
                   ▼ GET /api/notes?limit=20&offset=0
              ┌──────────┐
              │  GATEWAY │  NotesController (AuthGuard + AbilityGuard)
              └────┬─────┘   - sets ownerId on create
                   │         - filters list by ownerId unless user is admin
                   │         - asserts CASL Note.read/update/delete per id
                   ▼
              ┌──────────┐
              │ NOTES MS │  @MessagePattern('notes.list/get/create/update/delete')
              └────┬─────┘
                   ▼
              DBStrategy (set/update/get/delete/list)
```

CASL rules added to `defineAbilitiesFor`:

```ts
if (user.role === 'admin') {
  can('manage', 'Note');
} else {
  can('read', 'Note', { ownerId: user.id });
  can('create', 'Note'); // ownerId enforced server-side
  can(['update', 'delete'], 'Note', { ownerId: user.id });
}
```

The gateway runs `ability.can('read', 'Note', loadedNote)` before returning. Client uses `<Can I="update" a="Note" this={note}>` to gate the edit button (UX only — server is still the source of truth).

## Gateway routes

```
GET    /api/notes                          AuthGuard
       ?limit=20&offset=0
                                           → 200 { items: Note[], total: number }
GET    /api/notes/:id                      AuthGuard
                                           → 200 Note, 404 if missing, 403 if not yours
POST   /api/notes                          AuthGuard + @CheckAbility('create', 'Note')
       body: { title, body }
                                           → 201 Note (ownerId from req.user)
PATCH  /api/notes/:id                      AuthGuard + ownership check
       body: { title?, body? }
                                           → 200 Note (updatedAt bumped)
DELETE /api/notes/:id                      AuthGuard + ownership check
                                           → 204
```

All endpoints behind the default global throttler. No magic — pure CRUD.

## MS

```ts
@Controller()
export class NotesController {
  constructor(@Inject('DBStrategy') private readonly db: DBStrategy) {}

  @MessagePattern('notes.list')
  list(@Payload() p: { ownerId: string | null; limit: number; offset: number }) {
    const where = p.ownerId ? [{ field: 'ownerId', op: '==', value: p.ownerId }] : undefined;
    return this.db.list<Note>('notes', {
      where,
      orderBy: { field: 'createdAt', direction: 'desc' },
      limit: p.limit,
    });
  }
  // … get, create, update, delete
}
```

Offset is applied client-side after `list` — `DBStrategy` doesn't have offset, only limit. Acceptable trade-off for sample-grade.

## Client templates

New route `/_dashboard/notes` (3 templates):

- **Index** — table of notes with title, snippet, date, edit + delete actions. Empty state. Pagination controls.
- **New / Edit dialog** — modal form (title, body). Uses `useDraft` (`@idevconn/use-draft`) so closing prompts on dirty state.
- **Delete confirmation** — destructive action confirm.

Components live in `apps/templates/client-{shadcn,antd,mui}/src/components/notes/` per the one-component-per-file rule. Each template ships its own:

- shadcn: shadcn `Table` + `Dialog` + `Form`
- antd: `Table` + `Modal` + `Form`
- MUI: `Table` + `Dialog` + custom form

Shared TanStack Query hooks live in `apps/templates/client-{shadcn,antd,mui}/src/queries/notes.ts`. Same shape across templates because the API is identical.

Sidebar nav item added to `LayoutSider` in all 3 templates.

## Out of scope

- Search / full-text — needs provider-specific text indexing.
- Tags / categories — second collection, doubles the surface for marginal demo value.
- Sharing notes between users — requires sharing model + invitations.
- Markdown rendering — `react-markdown` works fine but adds template-specific styling work.
- Attachments — would tie into `StorageStrategy` for fun, but adds upload + signed-URL UX surface; punt to Plan 10.1.
- Soft delete / undo — adds a `deletedAt` field + restore endpoint; not worth the complexity for a sample.

## Tests

- MS: `NotesController` against `FakeDBStrategy` — list/get/create/update/delete round-trips.
- Gateway: `NotesController` with mocked `NotesService` + an in-process `defineAbilitiesFor` — owner can read own, admin can read all, non-owner can't read other people's notes (403).
- E2E (smoke): login → create note → reload → see it → edit → delete. Runs in `client-shadcn-e2e` against the gateway with `FakeAuthStrategy` + `FakeDBStrategy`.
