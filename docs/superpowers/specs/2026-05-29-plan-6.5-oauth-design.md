# Plan 6.5 — OAuth Sign-In (Google + GitHub, Server-Mediated)

**Date:** 2026-05-29
**Status:** Approved design — implementation lives in `docs/superpowers/plans/2026-05-29-plan-6.5-oauth.md`.

## Goal

Add OAuth sign-in for Google and GitHub across icore's two auth providers. Client redirects to the gateway → gateway hands the user off to the provider → provider returns the user to the gateway's callback → gateway exchanges the code for an `AuthSession` via the configured `AuthStrategy`. Client templates stay provider-agnostic.

## Why server-mediated

Two reasonable approaches:

- **(A) Client SDK per provider** — load Firebase JS or Supabase JS in the browser, call `signInWithPopup` / `signInWithOAuth` directly. Breaks the icore client-is-provider-agnostic invariant; templates would need different code per `AUTH_PROVIDER`.
- **(B) Server-mediated** — `AuthStrategy` exposes `startOAuth` / `completeOAuth`; the gateway owns the redirect flow. Client only knows about generic `/api/auth/oauth/:provider` endpoints. **Chosen.**

This matches the existing icore pattern: every provider-specific quirk lives behind the strategy interface, and the frontend never imports an SDK.

## AuthStrategy additions

```ts
export type OAuthProvider = 'google' | 'github';

export interface OAuthStartResult {
  redirectUrl: string; // browser is sent here
  state: string; // CSRF token, server keeps a copy in a cookie
}

export interface AuthStrategy {
  // ... existing methods
  startOAuth(provider: OAuthProvider, callbackUrl: string): Promise<OAuthStartResult>;
  completeOAuth(provider: OAuthProvider, code: string, state: string): Promise<AuthSession>;
}
```

Both methods are required (not optional) — every strategy must implement them. The `FakeAuthStrategy` ships a noop impl that round-trips a deterministic code for the contract suite.

## Concrete impls

### Supabase

- `startOAuth` calls `client.auth.signInWithOAuth({ provider, options: { redirectTo, skipBrowserRedirect: true } })` — returns `data.url` (provider URL with `state` baked in by Supabase). Strategy generates its own `state` token, stashes a `{state → provider}` map, and rewrites the URL's `state` param.
- `completeOAuth` calls `client.auth.exchangeCodeForSession(code)`.

### Firebase

- `startOAuth` — build `https://accounts.google.com/o/oauth2/v2/auth?…` (Google) or `https://github.com/login/oauth/authorize?…` (GitHub) directly. Each provider needs its own `clientId` + scopes from env (`OAUTH_GOOGLE_CLIENT_ID`, `OAUTH_GITHUB_CLIENT_ID`, …).
- `completeOAuth` — exchange the code with the provider's token endpoint for an `idToken`, then call Identity Toolkit `accounts:signInWithIdp` to mint Firebase tokens.

This is more code than Supabase because Firebase doesn't have a single "start OAuth" REST surface — but the strategy interface hides the difference from the gateway.

## Gateway routes

```
GET  /api/auth/oauth/:provider                  → 302 to provider redirectUrl (sets state cookie)
GET  /api/auth/oauth/:provider/callback?code&state → 302 to ${CLIENT_ORIGIN}/auth/oauth/callback#token=…
                                                     OR 200 JSON for SPA fetch flow
```

The redirect-to-fragment approach lets the SPA pick up the session without exposing the code to JS. The `state` cookie is `HttpOnly + Secure + SameSite=Lax` and cleared after verify.

Both routes are `@Public()` and inside the existing `auth-burst` throttle group.

## Client templates

`/login` adds:

- "Continue with Google" button
- "Continue with GitHub" button
- Both kick off `window.location.assign('/api/auth/oauth/google')`.

New `/auth/oauth/callback` route reads the URL fragment (`#accessToken=…&refreshToken=…&userId=…&email=…`), populates `useAuthStore.setAuth`, redirects to `/_dashboard/dashboard`. Same shape across all 3 templates; library-specific buttons (shadcn Button with icon, antd Button with `GoogleOutlined`, MUI Button with `<GoogleIcon>`).

## Identity merge — out of scope

If the OAuth identity's email matches an existing password user, the providers handle merging differently. Plan 6.5 ships whatever the provider does by default (Supabase merges, Firebase keeps separate accounts). Explicit account linking lands as Plan 6.6 if asked.

## Tests

- `runAuthContract` gains 2 cases (start + complete round-trip, complete rejects bad state). Helper hook (`getOAuthCode(strategy, provider)`) parallels `getMagicLinkToken`.
- Gateway controller test: state cookie set on start, verified on callback.
- Mock provider servers in `auth-firebase` / `auth-supabase` tests — no real HTTP.

## Out of scope

- Apple / Microsoft / Discord / etc. — same architecture, additive per provider.
- PKCE — server-mediated already protects the code; PKCE is a client-SDK concern.
- Account linking — Plan 6.6.
- "Sign in with one tap" / Google One Tap — front-end SDK, requires the client-SDK approach.
