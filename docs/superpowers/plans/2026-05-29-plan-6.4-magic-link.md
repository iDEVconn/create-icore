# Plan 6.4: Magic-Link Email Sign-In (Passwordless)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add passwordless email-link sign-in across the icore stack. User types email → server emits a one-time link via provider → user clicks link → arrives back at the app → callback exchanges the token for a session. Works identically for `AUTH_PROVIDER=supabase` and `AUTH_PROVIDER=firebase` because both providers expose a server-callable REST surface (Supabase `signInWithOtp` + Firebase Identity Toolkit `sendOobCode` / `signInWithEmailLink`). Client templates stay provider-agnostic; only the gateway + auth MS know which provider is configured.

**Scope:** Magic-link only. OAuth (Google, GitHub) is deferred to **Plan 6.5** because it requires client-side provider SDK + redirect callbacks, which is a separate architectural decision.

**Architecture:**

```
                 ┌────────────────────────────────────────────────────┐
                 │  CLIENT TEMPLATE (shadcn / antd / mui)             │
                 │                                                     │
                 │  /login          /auth/callback?token=…             │
                 │     │                  │                            │
                 │     ▼                  ▼                            │
                 │  POST              POST                             │
                 │  /api/auth/        /api/auth/                       │
                 │  magic-link        magic-link/verify                │
                 │     │                  │                            │
                 └─────┼──────────────────┼────────────────────────────┘
                       │                  │
                       ▼                  ▼
                 ┌────────────────────────────────────────────────────┐
                 │  GATEWAY (apps/api)                                 │
                 │  - AuthController.requestMagicLink                  │
                 │  - AuthController.verifyMagicLink                   │
                 └────────────┬───────────────────────────────────────┘
                              │ TCP / Redis / NATS
                              ▼
                 ┌────────────────────────────────────────────────────┐
                 │  AUTH MS (apps/microservices/auth)                  │
                 │  @MessagePattern('auth.magicLink.send')             │
                 │  @MessagePattern('auth.magicLink.verify')           │
                 │     │                                                │
                 │     ▼                                                │
                 │  AuthStrategy.sendMagicLink(email, callbackUrl)     │
                 │  AuthStrategy.verifyMagicLink(token)                │
                 └─────────────────────────────────────────────────────┘
```

Both methods land on the `AuthStrategy` interface as optional. Concrete strategies implement them; the in-memory `FakeAuthStrategy` ships a noop-style impl for the contract suite + the controller tests.

**Tech Stack:** No new client deps. Server-side: `@supabase/supabase-js` (already installed) — `auth.signInWithOtp` + `auth.verifyOtp`. Firebase: existing `HttpIdentityToolkitClient` extended with `sendOobCode` + `signInWithEmailLink` endpoints.

**Source spec:** `docs/superpowers/specs/2026-05-28-icore-design.md` Non-Goals section noted magic-link as out-of-scope for v0.1; Plan 6.4 promotes it to scope for the next minor.

## Task 2: Implement magic-link in `SupabaseAuthStrategy`

Supabase JS exposes:

- `client.auth.signInWithOtp({ email, options: { emailRedirectTo } })` — sends the link
- `client.auth.verifyOtp({ token_hash, type: 'magiclink' })` — exchanges OTP for session

Edit `libs/auth-strategies/supabase/src/lib/supabase-auth.strategy.ts`:

```ts
async sendMagicLink(req: MagicLinkRequest): Promise<void> {
  const { error } = await this.client.auth.signInWithOtp({
    email: req.email,
    options: { emailRedirectTo: req.callbackUrl },
  });
  if (error) throw new Error(error.message);
}

async verifyMagicLink(token: string): Promise<AuthSession> {
  const { data, error } = await this.client.auth.verifyOtp({
    type: 'magiclink',
    token_hash: token,
  });
  if (error || !data.session) throw new Error(error?.message ?? 'invalid_magic_link');
  return this.toSession(data.session);
}
```

Extend `mock-supabase.ts` to model `signInWithOtp` + `verifyOtp({ type: 'magiclink' })`:

- `signInWithOtp({ email })` — find-or-create user, generate token hash, store `{tokenHash → uid}`, return `{ data: {}, error: null }`.
- `verifyOtp({ type: 'magiclink', token_hash })` — look up the hash, return session + clear the hash.

Run `nx test auth-supabase` → 9 → 11 contract cases pass.

## Task 3: Implement magic-link in `FirebaseAuthStrategy`

Firebase Identity Toolkit REST:

- `accounts:sendOobCode` — sends the email link (`requestType: 'EMAIL_SIGNIN'`)
- `accounts:signInWithEmailLink` — exchanges the `oobCode` for an idToken + refreshToken

Extend `HttpIdentityToolkitClient` (`libs/auth-strategies/firebase/src/lib/identity-toolkit.client.ts`):

```ts
async sendOobCode(opts: { email: string; continueUrl: string }): Promise<void> {
  await this.post('https://identitytoolkit.googleapis.com/v1/accounts:sendOobCode', {
    requestType: 'EMAIL_SIGNIN',
    email: opts.email,
    continueUrl: opts.continueUrl,
  });
}

async signInWithEmailLink(opts: { email: string; oobCode: string }): Promise<IdentityToolkitSignInResponse> {
  return this.post('https://identitytoolkit.googleapis.com/v1/accounts:signInWithEmailLink', {
    email: opts.email,
    oobCode: opts.oobCode,
  });
}
```

`FirebaseAuthStrategy.sendMagicLink` → `sendOobCode`. `verifyMagicLink` is tricky: Firebase's `signInWithEmailLink` expects BOTH the email AND the `oobCode`. Our contract gives only the token — so we encode `email|oobCode` server-side via a base64 wrapper.

Pragmatic: encode `email + oobCode` as a single opaque token at `sendMagicLink` time and decode at `verifyMagicLink`:

```ts
async sendMagicLink(req: MagicLinkRequest): Promise<void> {
  const wrappedCallback = `${req.callbackUrl}?email=${encodeURIComponent(req.email)}`;
  await this.identityToolkit.sendOobCode({ email: req.email, continueUrl: wrappedCallback });
}

async verifyMagicLink(token: string): Promise<AuthSession> {
  // Token shape: base64(email):oobCode  (set by the gateway when it builds
  // the callback URL — see Plan 6.4 Task 4 for the wrap/unwrap pattern.)
  const [emailB64, oobCode] = token.split(':');
  if (!emailB64 || !oobCode) throw new Error('invalid_magic_link_token');
  const email = Buffer.from(emailB64, 'base64').toString('utf8');
  const res = await this.identityToolkit.signInWithEmailLink({ email, oobCode });
  return {
    accessToken: res.idToken,
    refreshToken: res.refreshToken,
    expiresIn: Number(res.expiresIn),
    user: { id: res.localId, email: res.email },
  };
}
```

Extend the firebase mock — `sendOobCode` records `{email → oobCode}`, `signInWithEmailLink` looks up + returns a session.

Run `nx test auth-firebase` → 9 → 11 contract cases pass.

**Commit:** `feat(auth-firebase): magic-link send/verify via Identity Toolkit sendOobCode + signInWithEmailLink`

### Step 2: `/auth/callback` route

New file `apps/templates/client-{shadcn,antd,mui}/src/routes/auth.callback.tsx`:

```tsx
import { createFileRoute, useNavigate } from '@tanstack/react-router';
import { useEffect, useState } from 'react';
import { useAuthStore, useNotify } from '@icore/template-shared';
import { api } from '../main';

function CallbackPage() {
  const navigate = useNavigate();
  const notify = useNotify();
  const setAuth = useAuthStore((s) => s.setAuth);
  const [status, setStatus] = useState<'verifying' | 'done' | 'error'>('verifying');

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const token = params.get('token') ?? params.get('token_hash') ?? params.get('oobCode');
    if (!token) {
      setStatus('error');
      notify.error('Missing magic-link token');
      return;
    }
    api<{
      accessToken: string;
      refreshToken: string;
      user: { id: string; email: string; role?: string };
    }>('/auth/magic-link/verify', { method: 'POST', body: JSON.stringify({ token }) })
      .then((session) => {
        setAuth(session);
        setStatus('done');
        navigate({ to: '/_dashboard/dashboard' });
      })
      .catch((err) => {
        setStatus('error');
        notify.error(err instanceof Error ? err.message : 'verification_failed');
      });
  }, []);

  // Library-specific spinner + message
}

export const Route = createFileRoute('/auth/callback')({ component: CallbackPage });
```

Each template ships its own visual (shadcn `<Loader />` icon + text, antd `<Spin />`, mui `<CircularProgress />`).
