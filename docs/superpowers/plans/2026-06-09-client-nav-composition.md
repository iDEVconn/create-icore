# Client — Config-Driven Nav Composition Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`).

**Goal:** Replace `removeNotesClientTail`'s per-UI sidebar regex (and i18n strip) with a config-driven nav: each `LayoutSider` reads a `nav.config.ts` (`NavItem[]`) via a per-UI icon map, and the generator writes `nav.config.ts` from a base (dashboard+profile) + chosen features' `clientNav` (notes).

**Architecture:** `nav.config.ts` lives per client template at `src/nav.config.ts` (imported as `@/nav.config`), committed with all entries (iCore demo). The 3 sidebars become static + data-driven (icon map per UI). `writeNavConfig(targetDir, opts)` overwrites `apps/client/src/nav.config.ts` after `selectClientTemplate`, including the notes entry only when `example=notes`. The notes i18n keys stay always (harmless). `removeNotesClientTail` is deleted.

**Tech Stack:** TS, React, TanStack Router, react-i18next, the create-icore generator. Tests: `yarn nx test create-icore`. Clients: `yarn nx build client-shadcn client-antd client-mui` (or `nx run-many -t build -p client-shadcn,client-antd,client-mui`).

> **Icon-map note:** each sidebar's `ICONS` map references all three icons (dashboard/notes/profile), so the icon imports are always "used" regardless of what `NAV_CONFIG` contains — no unused-import lint errors when notes is absent. `NAV_CONFIG` drives what renders; `ICONS` is a static lookup.

      <Menu
        mode="inline"

- [ ] **Step 3: Implement** — `tools/create-icore/src/manifest/wire-client.ts`

```ts
import { writeFile } from 'node:fs/promises';
import { join } from 'node:path';
import type { CreateIcoreOptions } from '../lib/options.js';
import { MANIFEST } from './index.js';

const NAV_CONFIG_FILE = 'apps/client/src/nav.config.ts';

interface NavEntry {
  to: string;
  labelKey: string;
  iconName: string;
  exact?: boolean;
}

const BASE_NAV: NavEntry[] = [
  { to: '/dashboard', labelKey: 'nav.dashboard', iconName: 'dashboard', exact: true },
];
const PROFILE_NAV: NavEntry = { to: '/profile', labelKey: 'nav.profile', iconName: 'profile' };

function renderEntry(n: NavEntry): string {
  const parts = [`to: '${n.to}'`, `labelKey: '${n.labelKey}'`, `iconName: '${n.iconName}'`];
  if (n.exact) parts.push('exact: true');
  return `  { ${parts.join(', ')} },`;
}

/** Generate apps/client/src/nav.config.ts: dashboard + chosen-feature nav + profile. */
export async function writeNavConfig(targetDir: string, opts: CreateIcoreOptions): Promise<void> {
  const entries: NavEntry[] = [...BASE_NAV];
  if (opts.example === 'notes' && MANIFEST.feature.notes.clientNav) {
    entries.push(MANIFEST.feature.notes.clientNav);
  }
  entries.push(PROFILE_NAV);

  const content =
    `/**\n` +
    ` * Sidebar navigation. UI-agnostic: each LayoutSider maps \`iconName\` to its own\n` +
    ` * icon library. Generated by create-icore.\n` +
    ` */\n` +
    `export interface NavItem {\n` +
    `  to: string;\n` +
    `  labelKey: string;\n` +
    `  iconName: 'dashboard' | 'notes' | 'profile';\n` +
    `  exact?: boolean;\n` +
    `}\n\n` +
    `export const NAV_CONFIG: NavItem[] = [\n` +
    entries.map(renderEntry).join('\n') +
    `\n];\n`;
  await writeFile(join(targetDir, NAV_CONFIG_FILE), content);
}
```

> `MANIFEST.feature.notes.clientNav` is typed `{route;labelKey;iconName;exact?}` (Task 3). The generated `NavItem.iconName` union must include every `iconName` any feature contributes — currently `'dashboard'|'notes'|'profile'`. If a future feature adds a new iconName, widen the union in BOTH this generator string AND the committed template `nav.config.ts` + the per-UI `ICONS` maps.
